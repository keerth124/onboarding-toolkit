# COT Customization Roadmap

This document captures the planned customization model for `conjur-onboard`.
The goal is to let customers adapt naming, policy layout, claims, and common
policy additions without forking each platform adapter or hand-editing generated
artifacts after every run.

## Problem

The current GitHub and Jenkins flows are intentionally opinionated:

- GitHub authenticators default to one authenticator per org.
- GitHub workload identity paths default to `data/github-apps/<org>`.
- GitHub workload host IDs default to the repository name.
- Jenkins workload identity paths default to `data/jenkins-apps/<controller>`.
- Conjur policy YAML is generated by Go code, not by reusable templates.

That works for the common path, but customers may need variations:

- Different authenticator naming conventions.
- Different workload policy branches, such as `data/github-apps` instead of
  `data/github-apps/<org>`.
- Custom JWT claims or additional enforced claims.
- Optional authenticator API fields not modeled by the toolkit yet.
- Additional groups or grants that are always attached to generated workloads.
- Future platform-specific naming conventions without changing the shared
  Conjur generation flow.

The customization model should support those cases while preserving the value
of generated `api/plan.json`, validation, apply, and rollback.

## Decisions

### Keep One GitHub Authenticator Per Org

GitHub should continue to default to one authenticator per org. Different orgs
may need different claims, different governance, or different rollout schedules.
A shared global GitHub authenticator would make those differences harder to
reason about.

Default authenticator name remains equivalent to:

```yaml
authenticator:
  name_template: "github-{{ .org }}"
```

Customers can override the convention:

```yaml
authenticator:
  name_template: "github-apps-{{ .org }}"
```

### JWT Authenticators Always Live Under `conjur/authn-jwt`

The authenticator family/location is not a broad customization point. For JWT,
the authenticator should always resolve under:

```text
conjur/authn-jwt/<authenticator-name>
```

The customizable part is the authenticator name, also referred to as the service
ID segment. All dependent artifacts must derive from the resolved name:

- Create-authenticator body.
- Workload annotation keys.
- Apps group ID.
- Group membership API paths.
- Self-hosted grant policy.
- Integration snippets.
- `NEXT_STEPS.md`.
- Rollback metadata.

Do not default to slash-containing authenticator names such as
`github-apps/acme`. They look like policy paths and may behave inconsistently
across APIs, CLI calls, URL encoding, and platform-side plugins.

### Host IDs Should Not Contain `/`

Host/workload IDs should not use `/` as part of the host ID itself. Slash
segments should represent policy path hierarchy, not a single host name.

Preferred shape:

```text
identity path: data/github-apps/acme
host id: onboarding-toolkit
full workload id: data/github-apps/acme/onboarding-toolkit
```

Avoid:

```text
identity path: data/github-apps
host id: acme/onboarding-toolkit
```

The parent policy branch provides namespace. The host ID should be a clean leaf
identifier.

### Identity Path Is Customizable

The policy branch where workloads live should be configurable.

Default recommendation:

```yaml
authenticator:
  identity_path_template: "data/github-apps/{{ .org }}"
```

Customer override for single-org or shared-branch deployments:

```yaml
authenticator:
  identity_path_template: "data/github-apps"
```

If the identity path omits org, the tool should detect duplicate host IDs and
fail before generating applyable artifacts.

Open product decision: the current recommendation is to keep the org branch as
the default because it prevents future repo-name collisions. If we decide that
the default should be `data/github-apps`, duplicate detection becomes mandatory
for the default path.

### Groups And Grants Should Be Typed First

Customers may want local admin, audit, or governance groups added to generated
workloads. The first implementation should support common group/grant use cases
as typed configuration, not raw policy snippets.

Recommended default group location:

```text
<identity_path>/<group-id>
```

Example with default GitHub org branch:

```text
data/github-apps/acme/admins
data/github-apps/acme/auditors
```

For the first slice, grants should support all generated workloads:

```yaml
policy_extensions:
  groups:
    - id: admins
      grant_to: all_workloads
    - id: auditors
      grant_to: all_workloads
```

Advanced selectors can come later.

### Typed Config Before Full Template Overrides

The first implementation should not start with arbitrary file templates. The
tool should first expose a typed customization file that feeds the existing
model and renderers. This protects validation, apply, rollback, and generated
metadata.

Template overrides should be added after the rendering context is stable.

## Proposed Config Shape

Example `cot-config.yml`:

```yaml
version: v1alpha1

authenticator:
  name_template: "github-{{ .org }}"
  identity_path_template: "data/github-apps/{{ .org }}"
  token_app_property: "repository"
  enforced_claims: []

workloads:
  host_id_template: "{{ .repo }}"

policy_extensions:
  groups:
    - id: admins
      grant_to: all_workloads
    - id: auditors
      grant_to: all_workloads

api:
  create_authenticator:
    merge: {}
```

Single-org/shared branch example:

```yaml
version: v1alpha1

authenticator:
  name_template: "github-{{ .org }}"
  identity_path_template: "data/github-apps"

workloads:
  host_id_template: "{{ .repo }}"
```

Org-specific authenticator name with default org branch:

```yaml
version: v1alpha1

authenticator:
  name_template: "github-apps-{{ .org }}"
  identity_path_template: "data/github-apps/{{ .org }}"

workloads:
  host_id_template: "{{ .repo }}"
```

## Architecture

Customization should be a cross-platform extension layer between platform
adapter defaults and Conjur artifact generation:

```text
platform discovery
  -> platform adapter defaults
  -> customization resolution
  -> normalized authenticator/workload model
  -> Conjur artifact rendering
  -> plan/validate/apply/rollback
```

The platform adapters should continue to own platform facts:

- How to discover resources.
- Which claims are available.
- Which default claim is safe.
- How a resource maps to a default workload.
- How to generate platform-side integration snippets.

The customization layer should own customer-specific rendering choices:

- Authenticator name template.
- Identity path template.
- Host ID template.
- Enforced claims.
- Common policy groups and grants.
- Optional API body merge fields.

The Conjur generator should still own:

- SaaS versus self-hosted endpoint selection.
- Policy load target selection.
- Workload policy rendering.
- Group membership REST bodies.
- Self-hosted grant fallback policy.
- `api/plan.json` metadata.
- Rollback metadata.

## Rendering Context

Use Go `text/template` with a small, documented context. Keep the variable set
stable across adapters where possible.

Common fields:

```text
.platform
.authenticator_name
.identity_path
.host_id
.source_id
.display_name
```

GitHub fields:

```text
.org
.repo
.repository
.repo_full_name
.default_branch
.visibility
```

Jenkins fields:

```text
.jenkins_controller
.jenkins_full_name
.folder
.job
.scope_type
```

Helper functions:

```text
safe
lower
upper
replace
default
trimPrefix
```

Guardrails:

- Authenticator name template must not resolve to an empty value.
- Authenticator name template must not contain `/` by default.
- Host ID template must not resolve to an empty value.
- Host ID template must not contain `/` by default.
- SaaS identity paths must resolve under `data`.
- Generated workload full paths must be unique.
- Generated annotation keys and group IDs must use the resolved authenticator
  name.

## Policy Extension Model

Typed policy extensions should be rendered into the workload policy where
possible.

For SaaS leaf-branch workload loads:

```yaml
- !host
  id: onboarding-toolkit
  annotations:
    authn-jwt/github-acme/repository: acme/onboarding-toolkit

- !group
  id: admins

- !grant
  role: !group admins
  members:
    - !host onboarding-toolkit
```

For self-hosted root-wrapper policy loads, the same extension should render
inside the policy body for the identity path.

Rollback behavior needs an explicit decision. Options:

- Treat custom groups/grants as manual-policy-review.
- Generate rollback delete operations for created groups.
- Generate rollback revoke operations for grants if the Conjur API supports the
  exact inverse operation for the target.

Recommended v1 behavior: mark custom groups/grants as manual-policy-review in
plan metadata, while still rolling back generated workload hosts and REST group
memberships.

## Template Override Model

Template overrides should be a later phase.

Eventually:

```yaml
templates:
  workload_policy: ./templates/workloads.yml.tmpl
  identity_branch_policy: ./templates/identity-branch.yml.tmpl
  self_hosted_authenticator_grants: ./templates/authenticator-grants.yml.tmpl
```

The toolkit should ship its own default templates, likely embedded in the Go
binary, and expose a command to copy them:

```text
conjur-onboard templates export --target ./templates
```

Even when templates are overridden, the typed model should remain the source of
truth for:

- Workload IDs.
- Authenticator name.
- Identity path.
- Apps group ID.
- Rollback metadata.
- Plan operation ordering.

## Code Changes

### New Package: `internal/customize`

Add a package for cross-platform customization:

- Config structs.
- YAML loading.
- Version validation.
- Template rendering.
- Safe-name helpers.
- Duplicate detection.
- Policy extension normalization.

Suggested types:

```go
type Config struct {
    Version          string
    Authenticator    AuthenticatorConfig
    Workloads        WorkloadConfig
    PolicyExtensions PolicyExtensionConfig
    API              APIConfig
}

type AuthenticatorConfig struct {
    NameTemplate         string
    IdentityPathTemplate string
    TokenAppProperty     string
    EnforcedClaims       []string
}

type WorkloadConfig struct {
    HostIDTemplate string
}

type PolicyExtensionConfig struct {
    Groups []GroupExtension
}
```

### `internal/platform`

Extend `platform.GenerationInput` to carry customization:

```go
Customization *customize.Config
```

If importing `internal/customize` from `internal/platform` creates an awkward
dependency, use a small platform-neutral struct or pass customization separately
through command-level generation config.

### Platform Adapters

Do not make adapters policy-template aware.

Update adapters only where needed to:

- Provide default template variables.
- Respect resolved authenticator name.
- Respect resolved identity path.
- Respect resolved host ID.

GitHub changes:

- Replace hardcoded `AuthenticatorName` behavior with resolver input.
- Replace hardcoded `IdentityPath` behavior with resolver input.
- Replace hardcoded `RepositoryHostID` behavior with resolver input.
- Stop rejecting supported enforced claims once the generator can render them.

Jenkins changes:

- Apply the same resolver pattern for controller identity path and workload host
  IDs.
- Preserve Jenkins-specific defaults such as `jenkins_full_name`.

### `internal/conjur`

Update artifact generation to accept policy extensions:

- Render extension groups in `02-workloads.yml`.
- Render extension grants to all generated workloads.
- Preserve current SaaS leaf-branch loading behavior.
- Preserve current self-hosted root-wrapper loading behavior.
- Include extension metadata in `plan.json`.
- Keep rollback metadata for generated workload hosts unchanged.

Update authenticator body generation to support a controlled JSON merge:

- Allow adding optional fields.
- Reject core-field overrides unless explicitly enabled later.

Core fields should include at least:

- `name`
- `type`
- `subtype`
- `enabled`
- `data.identity.identity_path`
- `data.identity.token_app_property`

### CLI Commands

Add a shared generate flag:

```text
--config <path>
```

Apply to:

- `github generate`
- `jenkins generate`
- `github express` later, if express should support customization

Do not add this flag to `discover`; discovery should remain platform factual.

### Validation And Preview

Before writing artifacts, render a preview model and validate:

- Authenticator name.
- Identity path.
- Apps group.
- Workload count.
- Duplicate host IDs.
- Duplicate full workload paths.
- Unsupported slashes in authenticator name or host ID.
- Missing template variables.

Consider writing this to:

```text
api/customization-preview.json
```

or include it in `api/plan.json`.

### Documentation

Update:

- `README.md`
  - Add customization overview.
  - Add examples for org branch and shared branch.
  - Document slash rules for authenticator names and host IDs.

- `docs/manual-testing.md`
  - Add manual tests for `--config`.
  - Add duplicate host ID failure test.
  - Add policy extension group/grant review.

- `docs/platform-modularity/README.md`
  - Explain customization layer placement.
  - Explain what adapters should and should not own.

- New user guide later:
  - `docs/customization.md`
  - Full config reference and examples.

## Implementation Slices

### Phase 1: Config Skeleton And Validation

Goal: load and validate `cot-config.yml` without changing behavior unless a
field is set.

Work:

- Add `internal/customize`.
- Add `--config` to `github generate` and `jenkins generate`.
- Parse `version: v1alpha1`.
- Validate unknown/unsupported values clearly.
- Add tests for missing file, invalid YAML, unsupported version, and valid
  empty config.

Acceptance:

- Existing generation output is unchanged without `--config`.
- `go test ./...` passes.

### Phase 2: Authenticator Name And Identity Path Templates

Goal: allow customer naming and branch layout changes.

Work:

- Resolve authenticator name from template.
- Resolve identity path from template.
- Enforce no `/` in authenticator name.
- Enforce SaaS identity path under `data`.
- Update annotation keys, apps group IDs, integration snippets, next steps, and
  plan metadata to use the resolved name.
- Add duplicate workload full path validation.

Acceptance:

- Config can produce `github-apps-<org>` authenticator names.
- Config can produce `data/github-apps` or `data/github-apps/<org>` identity
  paths.
- Rollback metadata points to the resolved workload paths.

### Phase 3: Workload Host ID Templates

Goal: allow naming convention changes while preventing slash-shaped host IDs.

Work:

- Resolve host IDs from template.
- Enforce no `/` in host IDs.
- Detect duplicate host IDs within an identity path.
- Keep full workload path as `<identity_path>/<host_id>`.
- Update GitHub and Jenkins adapter tests.

Acceptance:

- GitHub default remains repository-name host IDs.
- A custom config can prefix or suffix host IDs.
- Duplicate host IDs fail before artifact writing.

### Phase 4: Enforced Claims

Goal: support common customer-required enforced JWT claims.

Work:

- Allow supported enforced claims through generation.
- Ensure authenticator body includes `enforced_claims`.
- Document which claim values can be derived for workload annotations.
- Keep token app property support limited to claims the adapter can map safely.

Acceptance:

- GitHub can generate an authenticator body with enforced claims.
- Unsupported token app properties fail with actionable errors.

### Phase 5: Typed Policy Groups And Grants

Goal: add admin/audit/governance groups and grant all workloads to them.

Work:

- Add typed `policy_extensions.groups`.
- Render groups into workload policy.
- Render grants to all generated workloads.
- Support SaaS leaf-branch and self-hosted root-wrapper rendering.
- Mark extension rollback behavior as manual-policy-review in plan metadata.

Acceptance:

- Generated policy creates local groups under the identity path.
- Generated policy grants all generated hosts to those groups.
- Existing workload rollback remains functional.

### Phase 6: Authenticator API Merge

Goal: allow customers to add optional API fields not modeled yet.

Work:

- Add `api.create_authenticator.merge`.
- Deep-merge non-core fields into generated JSON.
- Reject core-field overrides.
- Add tests for merge success and rejected overrides.

Acceptance:

- Customers can add optional authenticator fields.
- Core generated fields remain controlled by the tool.

### Phase 7: Template Export And Overrides

Goal: introduce advanced template customization after the typed model is stable.

Work:

- Convert internal policy rendering to embedded default templates.
- Add `templates export`.
- Add template override paths in config.
- Render using the same stable context as typed config.
- Keep `plan.json` and rollback metadata model-driven.

Acceptance:

- Default templates reproduce current output.
- Customer override templates can customize policy shape.
- Apply and rollback remain driven by generated plan metadata.

## Open Questions

- Should the product default identity path remain `data/github-apps/<org>`, or
  should GitHub default to `data/github-apps` and rely on duplicate detection?
- Should policy extension groups be created under the identity path by default
  for every platform, or should each adapter provide a default group path?
- Should grants to customer-created central groups be supported in v1, or only
  local generated groups?
- Should `github express` support `--config` in the first slice, or should it
  stay a simple opinionated path until generate customization stabilizes?
- Should slash usage be permanently rejected for authenticator names and host
  IDs, or allowed later behind an explicit advanced flag?
